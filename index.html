半精度

大吞吐 -> 大的有效 batch -> 低噪声，并克服大 batch 带来的低噪声。

有效 batch 是指多卡的 batch 和，不是单模型的入口 batch size，一个有效 batch 一次参数更新。
有效 batch = 单模型的入口 batch size * 卡数
我们为了获得大吞吐，可以增加卡数，可以增加单模型的入口 batch size（即增加单卡的吞吐，单模型的入口 batch size 和单卡吞吐是一个先线性后收敛的关系）,不管采用哪种方式，都会导致大的有效 batch，导致低噪声。

克服大的有效 batch 带来的低噪声就需要一些算法的 trick 了：

  • Momentum 就是一种噪声，噪声就是一种 Momentum，噪声低了，可以通过 Momentum 来补偿噪声
  • 根据批量大小提高学习率，补偿噪声。理论表明，如果按 K 倍增加批量大小，则应按 K 的平方根倍提高学习率，以保持原有的变化。在大规模训练的实践中，线性调整学习速率（K 倍）也很常见。
    • 学习率预热
  • NovoGrad 优化器

Keras 指定的 batch size 是单模型的入口 batch size，并不是大 batch 的 size，通过乘以卡数，我们可以计算得出大 batch 的 size。

batch size 太大，噪声太低了，表现出来就是”发散了”，结果非常差！！batch size 大，然后 loss 又小，梯度就小，又没有足够的噪声冲出去，就被困在 sharp minimal(sharp minimal 处的 loss 相比 flat minimal 可能更小，但是 generation gap 却很大！)。

正如您在实验 1 中可能已经看到的那样，较大的批量数据可能导致准确性下降，尤其是网络的泛化能力下降。较大批量消磨了训练过程中的噪声。
此外，如果大 batch 的 size 较大，则每个 epoch 所执行的更新次数会更少，如果假设达到一个效果需要的更新次数一定，那么多 GPU 可能会延长训练时间。

epoch 是和 total 样本关联的。

假设：

  1. 单 GPU，batch size 25，耗时 t；
  2. total 样本 100

分析：

  • 单 GPU，batch size 25，一个 epoch 4 次更新；16 次更新，需要 4 个 epoch，耗时一共 $16 \times t$
  • 4 GPU，单模型的入口 batch size 25，大 batch size 100， 一次一个大 batch 更新一次，一个 epoch 1 次更新，耗时理想情况为 t，16 次更新，需要 16 个 epoch，耗时理想情况一共 $16 \times t$。

如果不理想，就延长了训练时间。

pb::GraphDef -> graph::Graph

一个 pb::GraphDef 对象仅仅是一个计算图定义，它定义了图的输入、图的输出，以及每一个操作的输入、输出，是一种逻辑上的图。但是在程序中，图的输入、图的输出，操作的输入(class Variable)、输出(class Variable)，以及操作(class Operation)本身都必须是一个个内存中的对象。graph::Graph 对象就是一个包罗上述各个对象的对象。。Graph::init 会将一个 pb::GraphDef 对象表征的计算图中的各个操作实例化成一个个 Operation 对象。

一个操作可能有输入，可能有输出，可能输入输出都有，输入、输出都必须是 Variable 对象，在实例化一个操作时，在 Operation::Operation(构造函数) 中 VariableManager 负责拿到一个操作的输入对象（一个或一些 Variable 的实例）并 attach 到该操作对象，负责创建出该操作的输出对象（一个/一些 Variable 的实例）并 attach 到该操作对象。

现在我们有了一个个操作对象及输入输出对象，是时候把我们的“数”加载进 ps_var 和 ext_var 操作所对应的 Variable 对象中了。

共识算法

更新于 2021-04-07

所谓共识就是让不同的 nodes agree on

  1. Total order broadcast 是可以实现这一点的，但是不够 fault tolerant。Single leader replication 可以实现 Total order broadcast，即通过 dictator 的方式实现了共识，每个 follower 都听话，大家都 agree on，其乐融融。但是如果 leader down 掉，且没有 automatic leader election 及 failover 的话，无人统帅就无法达成共识了。因此，需要 automatic leader election 及 failover，但是 automatic leader election 及 failover 的过程又需要共识，即需要使用共识去实现共识，conundrum。

  2. 2PC 可以用来实现分布式事务，也可以通过 coordinator 的协调实现 dictator 式的共识，但是也不够好。If the coordinator fails, in-doubt participants cannot decide what to do，无所适从。
