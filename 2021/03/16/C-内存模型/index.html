C++ Concurrency in Action: C++ 内存模型

更新于 2021-03-28

One of the most important features of the C++11 Standard is something most programmers won’t even notice. It’s not the new syntax features, nor is it the new library facilities, but the new multithreading-aware memory model.

  • memory_order_seq_cst 强制要求各个线程看到的 change order 之间零差异，也就是同一个 total order
  • memory_order_release/memory_order_acquire 允许各个线程看到的 change order 之间存在一定差异，但不能太大
  • 对于某一个原子变量存在全局一致的 change order，也就是各个线程看到的这一变量的 change order 是一致的，除此以外不再约束各个线程看到的 change order 之间的差异。

std::atomic


std::atomic 不仅仅用于获得原子操作，而且会用于指导编译器的优化。

用户自定义的原子

  • 连续内存块，可以使用 bitwise 的 memcpy() 进行复制
  • 不能有用户自定义的拷贝构造函数
  • 不能有用户自定义的移动构造函数
  • 不能有用户自定义的拷贝赋值运算符和移动赋值运算符
  • 不能有虚函数和虚基类

也就是必须 TriviallyCopyable，
auto is_trivially_copyable = std::is_trivially_copyable<A>::value;

为什么需要 compare_exchange_？

因为我们需要取出值，做运算然后回写，在回写时我们有可能发现当时取出的值已经失效了，因此运算的结果也就是无效的了，如果我们执迷不悟地回写运算结果，就会造成写丢失。

另外一种做法就是乐观锁，

如果不相等，说明有人改了 atomic_x，oldX 已经失效了，因此运算的结果 f(oldX) 也就是无效的了，就应该抛掉 f(oldX) ，不该将 f(oldX) 赋给 atomic_x，应该读最新的值重新计算，并且计算完之后再次 Check before Set。

compare_exchange_weak

如果两个值相等，按照道理，会将 desired 赋给 atomic_x 并返回 true，但是 weak 不可靠啊，weak 可能没有将 desired 赋给 atomic_x，并返回了 false。

但是如果将其放到 loop 中，false 会触发重试，下一次靠谱一次就可以了。也就说 weak 的不可靠也没啥影响了。

lock-free

我们平常说 atomic 是 lock-free，事实上 atomic 的内部实现不一定真正是 lock-free 的原子 CPU 指令，内部实现有可能是依赖于锁的，并且在有的架构上是加锁实现，有的架构上是不加锁实现。

std::atomic_flag一定是 lock-free 的。

std::atomic_flag is an atomic boolean type.
Unlike all specializations of std::atomic, it is guaranteed to be lock-free.
Unlike std::atomic<bool>, std::atomic_flag does not provide load or store operations.

多线程的话，顺序就很重要了，因为这个顺序可能被其他线程观察到，这个 reorder 可能会对其他线程的执行有影响。
