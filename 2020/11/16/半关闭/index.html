区别 close 和 shutdown


shutdown 并不改变 socket 的引用计数，shutdown 只会改变 socket 的 usability，也就是不会销毁 socket 相关的数据。只有通过 close 将引用计数降为 0，才会引起数据的删除。

Big difference between shutdown and close on a socket is the behavior when the socket is shared by other processes. A shutdown() affects all copies of the socket while close() affects only the file descriptor in one process.

读一个 shutdown read 的 socket，写一个 shutdown write 的 socket。

Before server application call read, client application calls close on this connection. And the FIN has arrived in server TCP, and server TCP has sent ACK. So at this point, if server application calls read on this socket, what will read return? Does it return the unread data in TCP receive buffer?

Yes.

Or does it just return 0 to indicate EOF, and the unread data is discarded?

No.

Before server application call read, client application calls shutdown with SHUT_WR on this connection. And the FIN has arrived in server TCP, and server TCP has sent ACK. So at this point, if server application calls read on this socket, what will read return? Does it return the unread data in TCP receive buffer? Or does it just return 0 to indicate EOF, and the unread data is discarded?

Same as above.

Before server application call read, client application just exit without calling either close or shutdown. And TCP in client host will send the FIN, this FIN has arrived in server TCP, and server TCP has sent ACK. So at this point, if server application calls read on this socket, what will read return? Does it return the unread data in TCP receive buffer? Or does it just return 0 to indicate EOF, and the unread data is discarded?

Same as above.

The only way the receive buffer can get cleared is by reading all the data, or if the server TCP receives a reset, i.e. an RST, which can be caused in various ways, or of course if the server application closes the socket without reading all the data, in which case it issues an RST if there was pending read data.

将 application 和 TCP/IP protocol stack in kernel 分开是理解一切的关键。
