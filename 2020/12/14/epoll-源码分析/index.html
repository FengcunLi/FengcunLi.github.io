Epoll 源码分析

更新于 2020-12-31


epoll_wait 负责把自己（本线程）放入 epfd 对应的文件的事件通知队列 （wait_queue_head_t wq）中，并自己安心地在 for loop 中休眠（堵塞），即负责等。
epoll_ctl 负责往红黑树中插入 epitem 节点，并把 epitem 实例、ep_poll_callback 事件回调方法组成一个 eppoll_entry 实例放入被监听文件的事件通知队列 （wait_queue_head_t *whead）中。epoll_ctl 的使命就结束了，即负责注册。回调 ep_poll_callback 负责过滤并 wake_up_locked(&ep->wq);激活 epfd 对应的文件的事件通知队列，从而让各个线程自行（竞争地）去取就绪事件。

epoll 的 api：

  • epoll_create
  • epoll_ctl
  • epoll_wait

epoll_create1

epoll_create 在 epoll 文件系统里建了个 file 结点。

epoll_wait

该方法参数中，epfd 为 epoll_create1 方法返回的 fd，events 为用户提供的 struct epoll_event 类型的数组，用于存放有监听事件发生的那些 epoll_event，maxevents 表示这个数组的长度，也表示 epoll_wait 方法最多可返回 maxevents 个事件就绪的 epoll_event。

该方法最后又调用了 ep_poll 方法，继续看下这个方法

该方法的主要操作有

  1. 判断是否有监听事件就绪，如果有，则直接调用 ep_send_events 方法把就绪对象拷贝到 events 里并返回。

  2. 如果没有，则先调用 init_waitqueue_entry 方法初始化 wait 变量，其中 current 参数为线程私有变量，其中包含线程相关的数据，通过这个变量也能找到相应的线程。

    一个 wait_queue_entry_t 实例对应一个线程/进程 （struct task_struct），default_wake_function 方法就是用来唤醒 p 变量对应的线程的。

  3. 初始化完 wait 变量之后，把它放到 eventpoll 的 wq 队列中。eventpoll 对应着一个文件描述符 epfd，epfd 对应着一个文件，一个文件对应着一个事件通知队列，也就是 eventpoll 的 wq，wq 中的每一项都是一个 wait_queue_entry_t 实例，都对应着一个线程/进程，这些线程就睡着这个队列上，直到 eventpoll 对应的文件上有事件就绪，这个文件依次通知 wq 中的线程。比如我们有两个线程对同一个 epfd 调用了 epoll_wait，它俩都在等 epfd 对应的文件上的就绪事件。

  4. 然后进入 for 循环，其逻辑为，检查是否有监听事件就绪，如果没有，则调用 schedule_hrtimeout_range 方法，使当前线程进入休眠状态。
  5. 当各种情况，比如 signal、timeout、监听事件发生，导致该线程被唤醒，则会进入下一次 for 循环，并检查监听事件是否就绪，如果就绪了，则跳出 for 循环，同时把 wait 变量从 eventpoll 的 wq 队列中移除。

  6. 调用 ep_send_events 方法把就绪事件的对象拷贝到用户提供的 events 数组中，然后返回。

    该方法的大体逻辑是，将 eventpoll 中的 rdllist 列表内容转移到 txlist 列表中，同时把 rdllist 列表置为空，现在 txlist 就持有了所有就绪事件对象。然后调用上面的回调方法 ep_send_events_proc，将txlist传入其中。

    该方法的操作大体为：

    1. 遍历 head 就绪列表中的所有对象，对其调用 ep_item_poll 方法，真正地去检查我们关心的那些事件是否存在。对于 tcp socket 对象，这个方法最终会调用 tcp_poll 方法。
    2. 如果有我们感兴趣的事件，则将该事件拷贝到用户 event 中。
    3. 如果该监听对象是 level-triggered 模式，则会把该对象再加入到就绪列表中，这也是 level-triggered 和 edge-triggered 在代码上表现出来的本质区别。
    4. 所有监听对象检查完毕后，此时满足条件的对象已经被拷贝到用户提供的 events 里，到这里方法就可以返回了。

epoll_ctl

  1. 如果 ep_op_has_event 返回 true，则拷贝用户提供的 event 到方法的私有变量里。

    这样，在该方法调用完成之后，用户的 epoll_event 对象还是可以重用的。

  2. 通过 epfd 找到 eventpoll 对应的文件。
  3. 通过 fd 找到要被监听的目标文件，比如 socket 文件。注意区别 f = fdget(epfd); 和 tf = fdget(fd); 对应的是两个不同的文件。
  4. 从 epfd 对应文件的 private_data 字段获取 eventpoll 实例。

  5. 调用 ep_find 方法，查找 eventpoll 实例中是否已经监听了目标文件，这个地方就会查找红黑树。

    epoll 通过 epoll_ctl 实现了增量形式(ADD，MODIFY，DELETE)，只有在调用 epoll_ctl 时才会发生数据的拷贝，而 while 中调用 epoll_wait 时并不发生数据的拷贝，这一点不像 select 和 poll，它俩每次 while 中调用堵塞时都会全量拷贝要监听的文件描述符和关注的事件，比较低效。正因为要实现增量形式，epoll 需要操作系统帮忙管理已经注册的要监听的文件描述符和关注的事件，并且要支持 epoll_ctl 的快速地 ADD，MODIFY，DELETE（插入/查找/删除）操作，于是采用了红黑树这一数据结构。

  6. 如果 op 是 EPOLL_CTL_ADD，则调用 ep_insert 方法，如果是 EPOLL_CTL_DEL，则调用 ep_remove 方法，如果是 EPOLL_CTL_MOD，则调用 ep_modify 方法，来执行进一步的操作。由代码我们还能看到，如果 op 是 EPOLL_CTL_ADD 或 EPOLL_CTL_MOD，内核会自动帮我们注册 POLLERR 和 POLLHUP 事件，这在 epoll_ctl 的 man 文档中也有提到。

ep_insert

该方法的大体操作为

  1. 为要被监听的目标文件分配一块内存，类型为 epitem，然后对其初始化。

  2. 初始化 ep_pqueue 实例。
    该实例的作用是，当 ep_ptable_queue_proc 方法被回调时，通过 ep_pqueue 实例可以拿到 epitem 实例。

  3. 调用 ep_item_poll 方法，将 epitem 等相关信息组成的实例，放到被监听文件的事件变动通知队列中，这样当被监听文件有事件变化时，就会调用该队列里各个实例的回调方法，看是否有其感兴趣的事件发生。

  4. 调用 ep_rbtree_insert 方法，把 epitem 实例放入到 eventpoll 实例的红黑树数据结构中。

  5. 如果 ep_item_poll 方法返回的事件中有我们感兴趣的事件，则将 epitem 实例放到 eventpoll 实例的 rdllink 列表中，然后调用 wake_up_locked 方法，通知那些因调用 epoll_wait 方法而阻塞的线程，有你感兴趣的事件发生，你可以在 rdllink 列表中查看了。

ep_item_poll

下面，我们再以 tcp socket 文件为例，具体看下 ep_item_poll 是如何做的。

如果是 tcp socket 类型的文件，ep_item_poll 方法最终会调用 tcp_poll 方法。

该方法的参数中，file 为 tcp socket 文件，wait 为 ep_item_poll 方法传过来的 poll_table 实例，该实例被上面的 init_poll_funcptr 方法初始化，使其_qproc 字段指向 ep_ptable_queue_proc 方法，这个方法一会会被用到。

该方法调用了 sock_poll_wait 方法，传入一些参数。其中 sk_sleep(sk) 参数(wait_queue_head_t *)是 tcp socket 事件变动通知队列。

sock_poll_wait 方法又调用了 poll_wait 方法。

poll_wait 方法又调用了 poll_table 中的_qproc 字段指向的方法，即上面提到的 ep_ptable_queue_proc 方法。

ep_ptable_queue_proc 方法的作用就是把 epitem 实例、ep_poll_callback 事件回调方法等，组成一个实例 eppoll_entry，然后添加到 whead 指向的队列中，即 tcp socket 的 sk_sleep(sk) 事件通知队列。

这样，当 tcp socket 有事件发生时，就会回调 ep_poll_callback 方法，该方法会根据该事件是否是我们感兴趣的事件，决定是否唤醒因调用 epoll_wait 而阻塞的线程。

该方法中的参数 key 表示的就是发生的事件，ep_poll_callback 方法先检查 key 中是否包含我们感兴趣的事件，如果包含，则将 epitem 实例添加到 eventpoll 的 rdllink 队列中，然后再调用 wake_up_locked 方法，将那些因调用 epoll_wait 而阻塞的线程唤醒，告知它们可以到 eventpoll 实例的 rdllink 队列中去查看，有哪些监听文件已经发生了我们感兴趣的事件。

在调用完 sock_poll_wait 方法之后，tcp_poll 方法会检查当前已经就绪的事件。

当 listen socket 的 accept 队列不为空时，该方法会返回 POLLIN 事件，即通知应用层可以 accept 了。

总结

epoll_wait 负责把自己（本线程）放入 epfd 对应的文件的事件通知队列 （wait_queue_head_t wq）中，并自己安心地在 for loop 中休眠（堵塞），即负责等。
epoll_ctl 负责往红黑树中插入 epitem 节点，并把 epitem 实例、ep_poll_callback 事件回调方法组成一个 eppoll_entry 实例放入被监听文件的事件通知队列 （wait_queue_head_t *whead）中。epoll_ctl 的使命就结束了，即负责注册。回调 ep_poll_callback 负责过滤并 wake_up_locked(&ep->wq);激活 epfd 对应的文件的事件通知队列，从而让各个线程自行（竞争地）去取就绪事件。
